# AGENTS.md

This file is the canonical source of truth for any AI agent working on this Astro blog project.

## Project Overview

Personal portfolio and blog built with Astro, React, and TailwindCSS v4. Blog posts are Markdown files in `src/content/blog/`. The site is deployed via Vercel.

## Setup Commands

- Install dependencies: `pnpm install`
- Start dev server: `pnpm dev`
- Build for production: `pnpm build`
- Preview production build: `pnpm preview`

Always use `pnpm`, never `npm` or `yarn`.

## Project Structure

```
src/
  content/
    blog/           # All blog posts live here as .md files
  pages/            # Astro page routes
  components/       # Reusable Astro and React components
  layouts/          # Page layouts
public/
  images/blog/      # Blog cover images (e.g. post13_img0.png)
```

## Blog Post Format

Every blog post in `src/content/blog/` must include this frontmatter:

```yaml
---
title: "Post Title Here"
description: "A short, compelling description."
author: Lenin AGC
date: YYYY-MM-DD HH:MM:SS
cover: /images/blog/postNN_img0.png
categories: [Category1, Category2]
tags: [tag1, tag2, tag3]
prev:
  text: "Previous Post Title"
  link: "/blog/previous-post-slug"
next: false
---
```

- `prev` and `next` link to adjacent posts. Set to `false` if there is no adjacent post.
- Cover images go in `public/images/blog/` and follow the naming pattern `postNN_img0.png`.
- The `date` field uses the format `YYYY-MM-DD HH:MM:SS`.

## Writing Style

This is the most important section. All written content must sound like a real person wrote it, not like it was generated by an LLM. Follow these rules strictly.

### Voice and Tone

- Write in first person. The author is Lenin AGC.
- Be conversational, direct, and opinionated. Have reactions to the things you're describing.
- Use contractions naturally ("it's", "don't", "you're").
- Vary sentence length. Short sentences hit harder. Then follow up with something longer that gives more detail or context.
- Acknowledge complexity and mixed feelings when they exist. Not everything is amazing.

### Humanizer Skill (Required)

Before finalizing any blog post, apply the humanizer skill located at `.agents/skills/humanizer/SKILL.md`. This skill defines 28 AI-pattern detectors and 560+ flagged vocabulary terms.

Key rules from the humanizer:

1. **Ban Tier 1 vocabulary**: do not use words like "delve", "tapestry", "vibrant", "crucial", "robust", "seamless", "groundbreaking", "leverage", "transformative", "paramount", "multifaceted", "myriad", "cornerstone", "reimagine", "empower", "catalyst", "invaluable", "bustling", "nestled", "realm", "unpack", "deep dive", "actionable", "impactful", or "learnings".
2. **Kill filler phrases**: "In order to" becomes "to". "Due to the fact that" becomes "because". "It is important to note that" becomes just saying it.
3. **No sycophancy or chatbot artifacts**: never write "Great question!", "I hope this helps!", or "Let me know if you need anything else."
4. **No generic conclusions**: never end with "The future looks bright", "Exciting times lie ahead", or similar empty wrap-ups.
5. **Vary sentence rhythm**: humans write in bursts. AI writes metronomically. Mix short and long sentences.
6. **Have opinions**: react to facts, don't just report them.

### Forbidden Characters and Patterns

- **Never use the em-dash character** (the long dash: `\u2014`). Replace with commas, colons, semicolons, periods, or restructured sentences. The em-dash is a strong AI writing indicator and must not appear in any content.
- Do not overuse bold text. Use it for a key term on first introduction, not for mechanical emphasis on every other phrase.
- Do not use emoji in article body text.
- Avoid "It's not just X, it's Y" negative parallelism constructs.
- Avoid the "Rule of Three" pattern ("innovation, inspiration, and insights").

### Structure

- Open with a personal hook. Greet the reader casually.
- Use `##` headings to organize sections. Don't over-structure with too many `###` sub-headings.
- Prefer narrative paragraphs over bulleted lists. Lists are fine for genuinely list-like data (features, commands, config fields), but explanations should flow as prose.
- Include code snippets where they add value. Always use fenced code blocks with a language specifier.
- End with a personal reflection and an open question to the reader.

## Code Style

- Keep components focused and reusable.
